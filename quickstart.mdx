## Prerequisites

- **Node.js** 18+ (or Bun)
- **Git**
- Optional (for proving): **Rust toolchain** (for SP1/RISC0 projects)

## Step 1: Define the claim you want to prove

Pick a single, high-signal claim first (examples):

- **Human** (anti-bot)
- **Over 18**
- **Has an account on X / GitHub / email provider**

Your MVP should generate a proof locally and produce a verifiable attestation that a website can consume.

## Step 2: Extension “sidecar” architecture

Your browser extension should contain 3 core modules:

- **zkTLS fetcher**: uses zkTLS (e.g. TLSNotary) to fetch authentic Web2 data privately.
- **Prover**: runs a lightweight zkVM (SP1 or RISC Zero compiled to WASM) to produce ZK proofs locally.
- **Attestation client**: submits proofs for aggregation + verification to zkVerify and polls for an on-chain attestation.

## Step 3: Proving pipeline (recommended MVP)

Use a simple pipeline you can iterate on:

- **Input**: a zkTLS transcript (authentic web data) + user-selected claim.
- **Compute**: a zkVM program that checks the claim and outputs a commitment.
- **Proof**: generate a proof (Groth16 / FFLONK / zkVM proof).
- **Verify & attest**: aggregate + submit to zkVerify for **&lt;1s** verification, then receive an **on-chain attestation**.

## Step 4: Website integration (consumer)

On the website side, treat zkVerify as a **verification oracle**:

- Ask the user to run the extension for a claim.
- Receive an attestation reference (ID / hash / address).
- Query attestation status via your backend (or a client-safe API proxy).
- Gate access **without learning** the underlying private data.

## Summary

- Extension = private data collection + proof generation
- zkVerify = fast verification + aggregation + attestations
- Website = simple attestation checks, chain-agnostic UX
